Why Most Fabric Migrations Fail — And How We Made Ours Stick

Phanindra Gaddam·8 min read·Power BI Architect

I've watched organisations announce Fabric migrations with fanfare, only to quietly shelve them six months later. Having led a successful Denodo → Databricks → Microsoft Fabric migration at a global enterprise client, I can tell you exactly why most of them stall. It's almost never a technology problem.

The Four Reasons Migrations Fail

1. They migrate reports, not architecture

The most common mistake: teams take their existing mess — badly designed models, ungoverned datasets, report sprawl — and move it verbatim into Fabric. At our client, we used the migration as a forcing function to redesign the entire semantic layer from scratch against a Medallion Lakehouse architecture.

2. They underestimate data platform dependencies

Fabric doesn't exist in isolation. Ours sat downstream of Databricks, which replaced Denodo, which had replaced SSRS. Each layer had assumptions baked in that the Power BI layer had inherited and forgotten. We spent two weeks documenting what Denodo was doing that nobody had written down.

The data debt you ignore during migration will find you later. It shows up in the report layer as 'wrong numbers'.

3. They don't plan for Direct Lake fallback scenarios

Direct Lake is transformative — until it isn't. There are scenarios where the engine falls back to DirectQuery mode silently. We built our models to be Direct Lake-first but tested every measure under DirectQuery fallback conditions.

4. Governance comes after, not before

We built workspace design standards, naming conventions, and service principal governance before we migrated a single report. It added two weeks to planning. It saved months of firefighting.

What We Did Differently

  • Documented all upstream Denodo logic before touching Fabric
  • Designed the Medallion Lakehouse (Bronze → Silver → Gold) in Databricks first
  • Built governance standards before migrating reports
  • Rebuilt — not migrated — the semantic layer with Direct Lake + DirectQuery fallback testing
  • Trained 200+ users before switching off the old platform
💡 Key Takeaway: Treat a Fabric migration like a platform redesign, not a lift-and-shift. The extra planning time pays back in months of avoided rework.

Zero Throttling at Scale: Eliminating Power BI Bottlenecks Across 100+ Workspaces

Phanindra Gaddam·10 min read·Power BI Architect

When I joined a major Life Sciences client engagement, executive dashboards were timing out, dataset refreshes were failing overnight, and the business was losing confidence in the platform. Eighteen months later, we had zero throttling incidents across 100+ workspaces.

Diagnose Before Optimising

My diagnostic framework has three layers: Capacity layer (CU consumption, throttling events), Model layer (table sizes, cardinality, storage modes), and Query layer (slow measures, inefficient filter contexts).

You can't optimise what you haven't measured. The Capacity Metrics App is the starting point, not an optional add-on.

The Seven Levers

1. Incremental Refresh Policy Redesign

Roughly 40% of refresh failures come from poorly designed incremental refresh policies. At our client, we standardised on a 3-day incremental window with 2-year historical partitions. Refresh times dropped by over 60% on the largest models.

2. Storage Mode Strategy

Import for executive fact tables; Direct Lake for large Delta tables in Fabric; Aggregation tables for sub-second executive summaries. Not one-size-fits-all.

3. DAX Measure Audit

Use DAX Studio Server Timings to profile every executive dashboard measure. Usual suspects: CALCULATE with complex filters, SUMX over large tables, time intelligence triggering full scans.

4. Gateway Architecture and Scaling

Separate dedicated gateway clusters by workload type: overnight batch refreshes, live DirectQuery sources, admin API calls. Latency on DirectQuery reports dropped immediately after separation.

5. Pre-Staging for Executive Reports

Materialise aggregated results into a dedicated staging table refreshed overnight. The dashboard reads from a tiny pre-aggregated table at lightning speed.

6. Workspace Isolation and Capacity Tiers

Executive and mission-critical reports on dedicated Premium capacity. Dev/test on shared capacity. Prevents a poorly-written dataset from consuming production CUs.

7. Automated Monitoring

PowerShell monitoring that flagged datasets with >90 minute refresh times and alerted at 80% CU headroom. Problems caught at 3am by alerts, not at 9am by angry emails.

✅ Result: Zero unplanned executive dashboard outages for 12 consecutive months. ~55% average refresh time reduction. CU consumption stable through month-end peaks.

The Governance Framework That Scaled Across 100+ Power BI Workspaces

Phanindra Gaddam·7 min read·Power BI Architect

When I took ownership of the Power BI platform at a large Life Sciences client, there were no governance standards. Every team had created workspaces their own way. Building a framework that actually sticks is harder than it sounds. Here's what worked.

Why Most Frameworks Don't Stick

They fail for three reasons: too complex to follow, enforced only at deployment time, or they live in a document nobody reads. Our framework had one rule: it had to be automatable, auditable, and explainable in under two minutes.

The Five Pillars

1. Workspace Taxonomy

Four tiers: Development, Test, Production, Admin/Monitoring. Naming convention: [BU]-[Domain]-[Tier]. Example: FIN-FinancialReporting-PROD. When you have 100+ workspaces and need to find finance production at 11pm, naming saves you.

2. Service Principal Strategy

Eliminated shared user credentials from all automated processes. Every pipeline, refresh, and API call uses a dedicated service principal with scoped permissions, rotated on a schedule, documented in a living register.

3. RLS/OLS Design Standards

Security roles assigned via Entra security groups — never individual users. When someone changes teams, update their group membership, not 15 individual report permissions.

Assign security roles to groups, not individuals. This is the single biggest governance lever most teams are ignoring.

4. Lifecycle Management and Deployment Pipelines

Enforced DEV → TEST → PROD promotion via Fabric deployment pipelines. No direct edits to production. Every change creates an automatic audit trail.

5. Automated Compliance Monitoring

PowerShell scripts running nightly, checking every workspace against standards. Non-compliant workspaces flagged in a monitoring dashboard. Repeat offenders escalated to IT leadership.

💡 Key Insight: Governance that requires humans to check compliance will always decay. Automate the checking. Make compliance the path of least resistance.

The framework is still the organisation's standard today — years after I designed it. Does it outlast the person who built it? That's the real test.

Persona-Based BI Architecture: Separating Executive, Operational & Self-Service Layers

Phanindra Gaddam·9 min read·Power BI Architect

The single biggest architectural mistake in enterprise Power BI: one giant semantic model shared by executives, managers, analysts, and self-service users. It becomes a monster — slow, fragile, and serving nobody particularly well.

The Three Personas

Executive Layer

Executives need KPIs, trends, and exception reporting. Sub-second response. Zero complexity. Implementation: Import mode, pre-aggregated data, max 15 measures, refreshed nightly, never falls back to DirectQuery.

Operational Layer

Operational managers need current data, drill-through, and filtering. Tolerates slightly longer load. Uses Direct Lake or DirectQuery for near-real-time where needed.

Analyst / Self-Service Layer

Analysts need flexibility to write measures and explore. Give them the Lakehouse SQL endpoint. Critically — their exploration should never affect executive dashboard performance.

When you build one model for everyone, you optimise for nobody.

How We Implemented This at Our Client

  • Gold Lakehouse tables in Databricks (single source of truth)
  • Executive semantic model — Import, pre-aggregated, <15 measures, nightly refresh
  • Operational semantic model — Direct Lake, full grain, near-real-time
  • Analyst Lakehouse endpoint — SQL access for ad-hoc queries

Each layer: separate Fabric item, separate workspace, separate capacity allocation.

🎯 Try This: List every measure in your largest model. Tag each: Executive, Operational, or Analyst. If more than 30% are Analyst-level measures in an Executive report — you have a persona problem.

After implementing this, executive dashboard load times dropped to under 2 seconds.

Direct Lake vs Import vs Live Connect: The Decision Framework

Phanindra Gaddam·6 min read·Power BI Architect

After designing semantic models across Fabric, Databricks, Snowflake, and SQL Server, I've developed a consistent decision framework. Here it is.

ModeBest ForAvoid When
ImportExecutive dashboards, complex DAX, <1B rowsReal-time requirements, very large tables
Direct LakeFabric Lakehouses, Delta tables, near-real-timeNon-Fabric sources, highly complex DAX
DirectQueryReal-time data, huge volumes, regulatoryComplex DAX, high concurrency without aggregations
Live ConnectReusing shared certified modelsWhen you need custom measures

The Four Questions I Ask

1. How fresh does the data need to be?

Within minutes? Import is off the table for large models. Within an hour or nightly? Import with incremental refresh is usually best. Real-time? DirectQuery or Direct Lake with near-real-time refresh.

2. Where does the data live?

Direct Lake is only available for data in Microsoft Fabric Lakehouses (OneLake). If data is in Databricks, Snowflake, or on-premises SQL Server — Direct Lake is not an option. This eliminates the choice for many hybrid architectures.

3. How complex is your DAX?

Import handles the most complex DAX. Direct Lake handles most standard patterns but some complex filter contexts trigger DirectQuery fallback. Test heavily before committing.

4. What is the concurrency profile?

Import is extremely efficient under high concurrency (data is in-memory). DirectQuery under high concurrency can hammer your source system.

There is no universally 'best' connectivity mode. Only the right mode for your specific data volume, freshness requirement, DAX complexity, and concurrency profile.
🔧 My Default for New Fabric Projects: Direct Lake for operational models on Fabric Lakehouses. Import with incremental refresh for executive summary models.

FUAM: The Tenant Monitoring Tool Every Fabric Admin Needs

Phanindra Gaddam·12 min read·Power BI Architect

If you're a Power BI or Fabric admin, you're probably monitoring across five different places: Admin Portal, Capacity Metrics App, Workspace Monitoring, Purview Hub, Usage Metrics Report. That fragmentation is exactly what FUAM solves.

What FUAM Does

Free, open-source, built entirely on Microsoft Fabric. Pulls from Power BI REST APIs into a Fabric Lakehouse, surfaces through a Power BI report. One place to see:

  • Every workspace, capacity, and user in your tenant
  • Activity logs — who opened which report, when, how often
  • Full inventory of all reports, datasets, dashboards, and Fabric items via Scanner API
  • Tenant settings snapshots and change history
  • Capacity Refreshable telemetry and refresh history
FUAM fills a genuine gap that Microsoft hasn't addressed with a single first-party solution.

What Caught Me Out During Deployment

Service Principal Permissions Are Precise

Exactly two tenant admin settings needed: 'Service principals can use Fabric APIs' and 'Service principals can access read-only admin APIs'. Do not add other Entra ID API permissions — the docs warn this causes pipeline errors.

The Activity API Has Edge Cases

The API occasionally rejects the full 30-day window. If Load_Activities fails, set activity_days_in_scope to 28 and re-run.

Semantic Model Refresh Order Matters

Schedule the semantic model refresh 30–60 minutes after the pipeline completes — not simultaneously. I learned this the hard way when the first scheduled run showed stale data.

Secret Expiry Breaks Everything

Set a calendar reminder before your Service Principal client secret expires. Update both Fabric connections when you renew it.

The Business Value

  • Which reports haven't been viewed in 90 days? (Report rationalisation)
  • Which datasets refresh but nobody uses? (Capacity waste)
  • Which workspaces have no governance applied? (Security audit)
  • What changed in tenant settings last month? (Change management)

We Killed the ‘Please Refresh My Dataset’ ServiceNow Ticket. Here’s How.

Phanindra Gaddam·~7 min read·Power BI Architect

Every enterprise Power BI administrator has a version of this story.

A business analyst has a critical board presentation in two hours. The overnight refresh ran at 2am. New source data landed at 9am. The report is showing yesterday's numbers. The analyst raises a ServiceNow ticket. The ticket routes to the BI team. Someone picks it up, navigates to the Power BI service, finds the right semantic model, clicks refresh, and responds to the ticket. Forty minutes gone. Sometimes more.

Multiply that by fifteen analysts, three times a week. That was the situation I was asked to fix.

The Root Cause Was Architecture, Not Process

The instinct is to fix the ticketing workflow — faster routing, SLA enforcement, a dedicated refresh queue. That is the wrong instinct. You are optimising a process that should not exist.

The real problem is that users in a governed Power BI environment cannot refresh Production semantic models directly. They do not have — and should not have — Contributor access to Production workspaces. That is correct governance design. But the side effect is that every ad-hoc refresh request becomes an IT dependency.

The question is not "how do we handle refresh tickets faster?" It is "how do we eliminate the need for a ticket entirely?"

The Solution: A Power App in Microsoft Teams

I built a canvas Power App that presents users with a curated dropdown — a list of semantic models they are authorised to refresh. They select the model, optionally add a note (e.g. "Month-end data landed, need refresh before 10am standup"), and hit one button: Refresh Now.

Behind the scenes, the button triggers a Power Automate flow. The flow calls the Power BI REST API's dataset refresh endpoint using a service principal scoped specifically to the workspaces and models it needs — nothing more. The flow then writes the request to a SharePoint list: who triggered it, which model, when, and what the status was.

The app is published as a tab in the relevant Microsoft Teams channels. Users never leave Teams. No browser navigation, no Power BI service login, no ticket, no wait. They click a button and the refresh starts within seconds.

The Refresh History Report

The service principal's API calls return refresh status. I built a companion Power BI report on top of the SharePoint tracking list and the Power BI refresh history endpoint — showing every refresh event across all monitored semantic models: scheduled and ad-hoc refreshes, who triggered them, duration, success or failure, and error messages where applicable.

This report serves three audiences simultaneously:

  • Business users — "Has my refresh completed yet?" answered instantly, without asking anyone.
  • BI administrators — full audit trail, and patterns: which models are triggered most ad-hoc (signal the scheduled cadence may need adjustment), which users are heaviest consumers.
  • IT governance — replaces manual ticket-counting with an accurate, always-current picture of refresh demand and platform load.

What Changed for the Business

Ad-hoc refresh tickets dropped to near zero within the first two weeks of rollout. The BI team reclaimed roughly three to four hours per week previously spent on manual refresh requests. For business users, a process that took between 20 minutes and two hours now takes under 60 seconds — self-served, from within Teams, without raising a single ticket.

The qualitative shift matters more than the metrics. Users feel trusted. They have agency over their own data. The BI team is no longer a bottleneck between the analyst and their report.

A Few Things Worth Noting If You Build This

Scope the service principal carefully

Use the Power BI REST API's datasets/{datasetId}/refreshes POST endpoint. Grant the service principal Dataset.ReadWrite.All at the workspace level — not the tenant level. Scope it as narrowly as possible.

Not every model should be in the dropdown

Expose only models where ad-hoc refresh is genuinely appropriate. Large models with complex refresh chains should still run through your pipeline orchestration. A one-click trigger for a 90-minute refresh a user does not understand is how you get capacity throttling complaints.

Build in a cooldown check

The Power Automate flow checks whether a refresh is already in progress before triggering a new one. Without that guard, enthusiastic users will queue five refreshes when the first one takes longer than expected.

💡 The unexpected bonus: The refresh history report became a governance conversation starter. When leadership sees a dataset triggered manually twelve times a week, that signals the scheduled cadence is wrong, the source data SLA is unreliable, or the business process needs to change. The data tells a story no ticket system ever could.

The Bigger Principle

The best BI platform features make the right thing easy and the wrong thing unnecessary. Removing the ServiceNow ticket from the ad-hoc refresh workflow did not reduce governance — it strengthened it, because every refresh is now tracked, audited, and visible in a way that manual ticket handling never was.

If your users are still raising tickets to refresh a dataset, you have not finished the architecture.

Your Fabric Workspaces Are Running Deprecated Spark Runtimes. Do You Know Which Ones?

Phanindra Gaddam·~8 min read·Power BI Architect

Most Fabric administrators don't know. Microsoft has been moving through its Spark runtime deprecation schedule. Runtime 1.1 (Spark 3.3) was retired and disabled on March 31, 2025. Runtime 1.2 (Spark 3.4) has its end-of-life clock ticking — disabled March 31, 2026. Runtime 1.3 (Spark 3.5, Delta Lake 3.2) is the current recommended GA version. Runtime 2.0 (Spark 4.0, Delta Lake 4.0) is now in Experimental Public Preview.

The problem is not the deprecation schedule — Microsoft communicates it clearly. The problem is visibility. In an enterprise with dozens or hundreds of Fabric workspaces, how do you actually know which workspaces are running which runtime? And how do you reach the right people to fix it?

I built a report to answer exactly that.

The Problem with Manual Runtime Checks

The straightforward approach: open each workspace, navigate to Workspace Settings → Data Engineering / Science → Spark Settings, check the version. In a small environment that is manageable. In an enterprise with 100+ workspaces across multiple business units, it is not — it is an afternoon of clicking, a spreadsheet that goes stale immediately, and no systematic way to act on what you find.

The other challenge is ownership. Even if you identify that a workspace is still on Runtime 1.2, you need to find the admin and communicate the urgency without sending a generic blast email that everyone ignores.

The Solution: Fabric Admin API + Power BI Report

The Fabric Admin REST API exposes workspace-level settings including the configured Spark runtime version. I used a Power Automate flow on a scheduled daily trigger to call the API, write results to a SharePoint list, and feed that into a Power BI report. The result: a live, always-current view of every workspace in the tenant categorised by runtime status.

The report has three core views:

Runtime Health Overview

A summary showing workspace counts by runtime version: how many on current Runtime 1.3, how many on deprecated Runtime 1.2 (deadline: March 2026), and how many on the already-disabled Runtime 1.1. One glance tells the platform health story.

Workspace Drill-Down

Enterprise Gateway Management at Scale: Installation, Clustering, Load Balancing, Policy Control, and the Logging You Cannot Afford to Skip

Phanindra Gaddam·~12 min read·Power BI Architect

Gateway management is one of those disciplines that enterprise Power BI architects routinely underestimate — right up until a production refresh fails at 7am and you have no idea which node in a three-member cluster is responsible, no logs configured to tell you, and no policy in place to stop someone from standing up a rogue personal gateway that bypasses everything you built.

This article covers what I have implemented across a large enterprise tenant: all four gateway types, cluster high-availability with load balancing configured beyond the defaults, a PowerShell-based policy restriction tool to control who can install gateways, and the logging architecture that makes it possible to diagnose problems before they escalate into incidents.

Understanding the Four Gateway Types — and When Each One Actually Belongs

Most Power BI documentation treats gateway selection as a simple decision. In practice, each type has meaningful architectural implications.

Personal Mode Gateway

Single-user, Power BI only, runs as an interactive application (not a Windows service), supports only Import mode scheduled refresh. No DirectQuery, no clustering, no shared data sources, no high availability. From a governance perspective, personal gateways are your highest risk surface. They run on laptops that go offline, are installed by individuals without IT oversight, and create a proliferation of ungoverned data source connections across your tenant. We restrict personal gateway installation via policy (covered below) and only permit them for approved self-service use cases with documented justification.

Standard (Enterprise) Mode Gateway

This is the workhorse for enterprise deployments. Runs as a Windows service, supports multiple users and data sources, handles DirectQuery and Live Connection alongside Import refresh, supports clustering and high availability, and is manageable via PowerShell and the Fabric Admin APIs. All production workloads run through standard mode gateways in our environment. The gateway service account runs under a dedicated domain service account — not the default local PBIEgwService account — so permissions are managed at the Active Directory level and auditable through standard IT processes.

VNet (Virtual Network) Data Gateway

Microsoft-managed, no installation required, designed for data sources within Azure Virtual Networks. Unlike the on-premises gateway, it does not require inbound firewall ports — traffic stays within the Azure network boundary. We use VNet gateways specifically for Azure-hosted data sources within private VNets: Azure SQL Databases with private endpoints, Azure Synapse workspaces on isolated VNets, and Databricks clusters with VNet injection. The key limitation is that VNet gateways have a fixed hardware configuration that cannot be scaled or changed — there is no equivalent to clustering. For high-throughput scenarios on VNet-isolated sources, we use multiple VNet gateway instances and distribute semantic models across them manually.

Personal Mode vs Standard: The Decision We Enforce

The rule we implemented is simple: any data source used by more than one person or in any Production workspace must use a standard mode gateway cluster. Personal gateways are blocked by tenant policy for all users except a specific approved group. This is enforced through PowerShell, which brings us to the next section.

Restricting Gateway Installation via PowerShell Policy

One of the most underused capabilities in the Power BI gateway management toolkit is the ability to enforce a tenant-level installation policy programmatically. Without this, any user in your organisation can install a standard or personal gateway on any machine — creating ungoverned data paths that bypass your security architecture entirely.

The DataGateway PowerShell module exposes Set-DataGatewayTenantPolicy, which allows a Fabric Administrator to restrict gateway installation to a specified list of approved users or groups:

# Restrict standard gateway installation to approved list only
Set-DataGatewayTenantPolicy -ResourceGatewayInstallPolicy Restricted

# Grant installation rights to the gateway admin group only
Set-DataGatewayInstaller -PrincipalObjectId <GroupObjectId> -GatewayType Resource

# Restrict personal gateway installation separately
Set-DataGatewayTenantPolicy -PersonalGatewayInstallPolicy Restricted

# Audit current policy
Get-DataGatewayTenantPolicy

# List all currently authorised gateway installers in the tenant
Get-DataGatewayInstaller -GatewayType Resource -Scope Organization

We run the audit cmdlet as part of a weekly governance check, alongside Get-DataGatewayCluster -Scope Organization to enumerate all gateways across the tenant. Any new gateway that appears outside the approved list triggers an investigation and, if ungoverned, is removed via Remove-DataGatewayCluster and its installer permissions revoked.

This is not hypothetical governance theatre. We have found personal gateways installed on developer workstations, connected directly to production SQL Server instances, with credentials stored under individual user accounts. The policy restriction prevents that from happening silently.

Gateway Clusters: High Availability Beyond the Defaults

Creating a gateway cluster is well-documented — install a standard gateway on a second server and join it to the existing cluster during registration. What is less documented is how to tune the cluster behaviour beyond what the Power BI service UI exposes.

The Default Load Balancing Behaviour

By default, when you enable "Distribute requests across all active gateways in this cluster" in the Power BI service, request distribution is random. That sounds fine until you have a cluster where Node A is on a 32-core server and Node B is on an 8-core VM that was provisioned as a temporary failover node and never upgraded. Random distribution in that scenario means 50% of requests go to an underspecified machine.

Tuning Load Balancing Through Config

The gateway cluster load balancing behaviour can be manipulated through the gateway configuration file Microsoft.PowerBI.DataMovement.Pipeline.GatewayCore.dll.config, located in C:\Program Files\On-premises data gateway\ on each node. The key parameters for throttling are:

<!-- CPU throttling threshold - gateway node excluded from
     load balancing pool when CPU exceeds this % -->
<setting name="CPUThreshold" serializeAs="String">
    <value>75</value>
</setting>

<!-- Memory throttling threshold in MB -->
<setting name="MemoryThreshold" serializeAs="String">
    <value>10240</value>
</setting>

<!-- Max concurrent queries per node -->
<setting name="ConcurrencyLimit" serializeAs="String">
    <value>30</value>
</setting>

By setting differentiated CPU and memory thresholds across nodes, you can create a weighted distribution: the high-spec nodes absorb traffic until they approach threshold, at which point the cluster spills over to secondary nodes. This is not true weighted round-robin, but it is a practical approximation when your cluster nodes have unequal capacity.

We also implement workload separation by running two distinct gateway clusters: one cluster dedicated to scheduled Import refresh (fewer, large, memory-intensive queries), and a second cluster for DirectQuery and Live Connection (many, small, latency-sensitive queries). These two workload profiles have completely different resource characteristics and should not compete on the same nodes.

The concurrency limit per gateway node is 30 by default. If all nodes in a cluster hit their concurrency ceiling simultaneously, the next request fails outright. Plan your cluster size based on your peak concurrent query count, not your average.

Important: Keep All Cluster Members on the Same Gateway Version

This is the most common self-inflicted cluster failure I have seen. When gateway updates roll out monthly, staggered patching is good practice — but leaving nodes on different major versions for more than a few days introduces unpredictable behaviour. Keep a strict policy: all nodes patched within 24 hours of each other, always.

Gateway Logging: The Discipline Most Teams Skip Until It Is Too Late

The gateway generates three standard log files, written to %LOCALAPPDATA%\Microsoft\On-premises data gateway\ on the service account's profile:

  • GatewayInfo.log — General operational events: service start/stop, cloud service registration, heartbeat, query routing decisions, cluster membership changes.
  • GatewayErrors.log — Error-level events: connection failures, credential errors, query timeouts, mashup engine errors, throttling events.
  • GatewayNetwork.log — Network-level events: Azure Relay connectivity, outbound port issues, proxy negotiation, TLS handshake failures.

By default (post-February 2023 installations), the gateway retains logs for 30 days up to a maximum of 5GB total, with individual log files capped at 100MB before rotating. These defaults are configured in Microsoft.PowerBI.EnterpriseGateway.exe.config.

The Performance Monitoring Logs (Structured Logging)

Beyond the standard operational logs, enabling the performance monitoring feature in Microsoft.PowerBI.DataMovement.Pipeline.GatewayCore.dll.config generates four additional structured log files designed for analysis in Power BI:

  • QueryExecutionReport — Per-query: gateway ID, request ID, data source type, query type (Import/DirectQuery/Power Apps/Power Automate), execution start time, total duration, data processing duration, success/failure. The QueryText field is base64-encoded for security.
  • QueryExecutionAggregation — Rolled up to 5-minute intervals by gateway, data source, query type, and success status. Used for trending without the noise of per-query detail.
  • SystemCounterAggregation — CPU and memory utilisation, aggregated at 5-minute intervals. Critical for capacity planning and throttling threshold calibration.
  • ErrorReport — Structured error events with full context, separate from the text-based GatewayErrors.log.

Microsoft provides a Gateway Performance PBI template (.pbit) that reads these structured files directly. I have extended this template significantly in our environment: adding cross-node comparison views for cluster analysis, overlaying refresh failure events from the Power BI service activity log, and alerting on sustained CPU above threshold for more than 15 minutes.

The Logging Tradeoff: What to Enable and When

ModeWhat Is CapturedProsCons
Standard logging (default)GatewayInfo, GatewayErrors, GatewayNetworkAlways-on, negligible overhead, 30-day retention, covers most diagnostic scenariosText format, no per-query timing, no structured aggregation, hard to query at scale

Additional (verbose) loggingMashup engine internals, full query strings, verbose traceEssential for deep debugging, credential issues, connector-level failures, data source-specific errorsSignificant disk growth (can be 10x standard), short useful retention period, only enable for active investigation, disable immediately after

The Practical Tradeoff Decision

Standard logging should always be on — it is essentially free and covers the majority of incident investigation scenarios. Performance monitoring logs should be permanently enabled on all standard mode production gateways; the overhead is minimal and the structured format is invaluable for capacity planning. Verbose additional logging is strictly break-glass: enable it, reproduce the issue, export the logs, disable it. Leaving verbose logging enabled on a busy gateway can fill a disk in hours.

What Logs Actually Tell You That Monitoring Cannot

The most useful thing gateway logs have told me is not what failed — it is why it failed and which layer it failed at. A refresh failure in the Power BI service can originate from five different places: the data source itself, the network path between gateway and data source, the gateway service process, the Azure Relay connection between gateway and Power BI cloud service, or the query itself. The GatewayErrors.log combined with GatewayNetwork.log tells you exactly which layer. Without those logs, you are guessing.

Three real patterns I have diagnosed exclusively through logs:

  • Credential cache stale failures — The gateway caches credentials for approximately 5 hours. When someone updates a data source credential in the Power BI service, refreshes continue failing until the cache expires. This shows up in GatewayErrors.log as repeated InvalidConnectionCredentials errors with timestamps that cluster exactly 5 hours after a credential change event — not immediately. Without log timestamps, this looks like a random refresh failure.
  • Azure Relay connectivity drops — GatewayNetwork.log records every Azure Relay reconnection event. A pattern of multiple reconnections per hour, combined with periods of GatewayInfo.log silence, indicates an upstream network issue between the gateway host and Azure — usually a proxy, firewall rule, or DNS resolution problem that IT networking can investigate with precision if you hand them the log timestamps.
  • Concurrency ceiling hits — When a gateway node hits its 30-query concurrency limit, subsequent queries fail immediately. This shows up as a spike of identical ConcurrencyLimitExceeded errors in GatewayErrors.log at a specific time. Cross-referencing with SystemCounterAggregation confirms whether this was a CPU-bound failure (throttling kicked in first) or a pure concurrency ceiling (CPU was fine, just too many parallel requests). The fix is different in each case.
💡 The Governance Principle: Gateway logs are not just a debugging tool — they are a governance artefact. Audit log data from GatewayInfo combined with query data from the performance monitoring logs tells you which data sources are queried most, which semantic models drive the heaviest gateway load, and whether your cluster sizing is appropriate for the next quarter's refresh schedule. Build a Power BI report on top of them. Collect them centrally from all nodes. Treat them the same way you treat your activity logs from the Power BI service.

Gateway management at enterprise scale is unglamorous work. It lives in config files, PowerShell sessions, Windows service accounts, and log directories that most BI professionals never think about until something breaks. Getting it right — typed policies, proper clustering, workload separation, always-on structured logging — is what separates a platform that handles 100+ workspace refresh orchestration reliably from one that generates support tickets at month-end.

Enterprise Gateway Architecture: Installation, Load Balancing, Policy Control and Why Logging Saves You at 2am

Phanindra Gaddam·~12 min read·Power BI Architect

Gateway administration is one of those disciplines that is easy to underestimate until something breaks at 7am on a Monday and an executive's dashboard is blank. I have spent significant time across our enterprise deploying, configuring, and hardening every gateway type Microsoft offers — Personal, Standard (Enterprise), and VNet — and building controls around them that most teams don't have. This article covers what I did, the config-level details that matter, and why logging deserves far more attention than most Power BI teams give it.

The Four Gateway Types: What They Are and When to Use Each

Microsoft currently offers three on-premises gateway modes and one managed cloud gateway, and they serve genuinely different purposes. Getting this wrong at the start creates technical debt that is painful to unwind.

Personal Gateway (On-Premises Personal Mode)

The Personal gateway runs as an interactive Windows application — not a service. It only operates when the installing user is logged in and the application is running. It supports a single user, works only with Power BI (not Power Automate or Power Apps), and supports only Import mode scheduled refreshes. No DirectQuery, no Live Connection, no clustering, no high availability.

In an enterprise context, Personal gateways are a governance liability. They proliferate silently — any Power BI user can install one without administrator approval. They run under personal user credentials, meaning when that person leaves the organisation, their gateway orphans every refresh job tied to it. They are invisible in the central Manage Connections and Gateways admin view unless you know to look.

The correct use case for a Personal gateway is an individual analyst refreshing a handful of Import models from their own machine for exploratory work. It has no place in a production data estate.

Standard Gateway (On-Premises Standard Mode / Enterprise)

The Standard gateway runs as a persistent Windows service under a dedicated service account — typically NT SERVICE\PBIEgwService or a domain service account you specify. It operates independently of any user being logged in. It supports multiple users, multiple data sources, DirectQuery, Live Connection, composite models, clustering, high availability, and works across Power BI, Power Automate, Power Apps, Azure Data Factory, and Azure Logic Apps.

This is what production environments should run. A single Standard gateway cluster can support up to 10 member nodes and up to 1,000 data sources per cluster. Installation requires local administrator rights on the host machine, and the service account needs appropriate permissions to reach the on-premises data sources it will query.

A critical installation decision: use a dedicated service account in Active Directory rather than the default NT Service account. This gives you audit trail capability — you can trace exactly which queries the gateway service account executed against your source systems — and credential rotation through standard AD processes.

VNet Gateway (Virtual Network Data Gateway)

The VNet gateway is fundamentally different from the other two: it is a Microsoft-managed service with no installation required. You provision it through the Power Platform Admin Centre, and Microsoft spins up containerised gateway instances inside your Azure Virtual Network. It supports Dataflow Gen2, Power BI Semantic Models, Platform Dataflows, and Paginated Reports.

The VNet gateway is the right answer when your data sources are cloud-hosted but behind Azure private endpoints or inside a VNet — Azure SQL with private link enabled, Azure Synapse behind a managed VNet, Azure Data Lake with restricted public access. The on-premises Standard gateway does not support private link scenarios; the VNet gateway does.

The trade-off: VNet gateways require Fabric or Power BI Premium capacity. They auto-pause after a configurable inactivity period (default 30 minutes, configurable up to 24 hours) and take 2-3 minutes to resume after auto-pause. For workloads that need immediate availability, this cold-start delay matters. VNet gateway clusters support a maximum of 3 members, compared to 10 for Standard.

Choosing the wrong gateway type is not just a technical mistake — it is a governance mistake. Personal gateways spread silently, VNet gateways add latency on cold starts, Standard gateways require infrastructure management. Know which you are deploying and why before you begin.

Load Balancing: What the UI Doesn't Tell You

The Power BI admin portal exposes a single toggle under gateway cluster settings: "Distribute requests across all active gateways in this cluster." Most administrators enable this toggle and consider load balancing done. That is only half the story.

How Default Load Balancing Actually Works

With the toggle enabled, the gateway cloud service randomly selects a member node for each incoming request. It is not round-robin — it is random selection with availability checking. If the selected member is at or above a throttling threshold, the service selects another member. If all members are throttled, the request fails. Requests always route to the primary member when the toggle is disabled, regardless of its load.

The Config File That Changes Everything

The real load balancing controls live in a file that most administrators have never opened:

C:\Program Files\On-premises data gateway\Microsoft.PowerBI.DataMovement.Pipeline.GatewayCore.dll.config

This XML configuration file exposes the throttling thresholds that control when the load balancer considers a node "overloaded" and routes to another member. The three key parameters are:

  • CPUUtilizationPercentageThreshold — CPU percentage at which the node is considered throttled. Default is 0 (disabled). Setting this to, say, 80 means any node above 80% CPU will be skipped by the load balancer.
  • MemoryUtilizationPercentageThreshold — Memory percentage threshold. Also defaults to 0 (disabled). Critical to set in environments with large Import model refreshes that consume significant RAM.
  • ConcurrentOperationCountThreshold — Maximum simultaneous queries a single node will accept before being considered throttled. This is enabled by default. Setting this correctly prevents one large batch job from consuming all concurrency slots on a node, blocking small DirectQuery requests.

These values must be set identically on every node in the cluster. They are not synchronised automatically. When we deployed our cluster, we used a PowerShell script to distribute the config file changes across all nodes after any update, ensuring consistency. A node with different thresholds from its cluster peers will behave unpredictably under load.

After changing these values, the gateway Windows service must be restarted on each node for the changes to take effect.

Offline Nodes: The Silent Killer

One behaviour that catches teams off-guard: if a cluster member goes offline without being explicitly disabled in the admin portal, the load balancer will still attempt to route requests to it before failing over to an available node. This adds latency to every request during the failover window. Best practice is to disable a node in the portal before taking it offline for maintenance, and re-enable it only after confirming the service is healthy.

Policy Restriction: Preventing Unauthorised Gateway Installations

In a large enterprise, the proliferation of Personal gateways is a real and persistent governance problem. Any user with local administrator rights on their workstation can install a Personal gateway and begin routing sensitive data through it — outside your monitoring, outside your security controls, outside your data loss prevention policies.

I built a PowerShell-based policy enforcement tool that addresses this at the machine level. The approach has two components:

Detection

The script queries the Active Directory tenant via the Power BI Admin REST API to enumerate all registered gateways, then cross-references against a whitelist of approved gateway hosts and service accounts. Any Personal gateway or Standard gateway registered from a non-approved host is flagged and reported. This runs as a scheduled task and feeds a Power BI report showing unauthorised installations in near-real-time.

Prevention

For machines where Personal gateways should never be installed, the script enforces a Windows Registry policy that blocks the gateway installer from completing registration. The gateway software can be downloaded and even installed locally, but the registration step — which is what creates the cloud-side gateway entry — fails with a policy violation error. Combined with software restriction policies that can be deployed via Group Policy, this creates a two-layer control.

For Standard gateway installations, the enforcement is organisational: only approved server machines in approved OU paths in Active Directory are permitted to register new Standard gateways. The PowerShell script monitors for out-of-policy registrations and can invoke an automated deregistration flow via the Power BI Admin API for any gateway that registers without going through the approved change management process.

💡 Why this matters: A Personal gateway running on a departing employee's laptop, connected to a production SQL Server, is a data exfiltration risk, a refresh reliability risk, and a compliance audit finding waiting to happen. Prevention is significantly cheaper than remediation.

Gateway Logging: The Most Underused Diagnostic Tool in Power BI

When a dataset refresh fails, the first thing most administrators do is check the Power BI Service error message. That message is almost always generic — "Refresh failed," "Data source error," "Query timed out." The real diagnostic information is in the gateway logs, and most teams have never looked at them.

The Three Default Log Files

The Standard gateway writes three log files by default, located at %LOCALAPPDATA%\Microsoft\On-premises data gateway\ under the service account's profile (typically C:\Windows\ServiceProfiles\PBIEgwService\AppData\Local\...):

  • GatewayInfo.log — General operational events: gateway start/stop, connection establishment, data source queries, refresh requests received. Verbose by default. New file created daily, up to 100MB per file, 5GB total retention, 30-day age limit.
  • GatewayErrors.log — Error events only: failed connections, credential failures, timeout events, data source unreachable. This is your first stop for any refresh failure investigation.
  • GatewayNetwork.log — Network-level events: Azure Relay connection status, outbound connectivity checks, proxy events. Critical when troubleshooting gateway-to-cloud connectivity rather than gateway-to-datasource connectivity.

Additional Logging: The Double-Edged Sword

Beyond the default logs, the gateway supports Additional logging, enabled via the gateway application's Diagnostics tab or by setting parameters in GatewayCore.dll.config:

  • QueryExecutionReportOn = True — Enables per-query structured logging including query execution time, data source type, query status (success/failure), and the base64-encoded query text.
  • SystemCounterReportOn = True — Enables CPU and memory counter logging at configurable aggregation intervals (default 5 minutes) for the gateway process, mashup engine, and system overall.

When enabled, these generate additional structured log files in a Report subfolder that can be fed directly into the Gateway Performance PBI template — a Microsoft-provided Power BI report template that visualises query throughput, execution times, error rates, and system resource usage across your gateway cluster.

The Logging Trade-off: Why You Should Not Leave It On Permanently

This is where most guidance oversimplifies. The Microsoft recommendation is to "temporarily enable Additional logging for troubleshooting." The reality is more nuanced:

The case for permanent additional logging: In a production environment with mission-critical refreshes, having continuous query-level telemetry means you can diagnose a 3am refresh failure in the morning with full context — exactly which query timed out, exactly how long it ran, exactly what the resource utilisation was at that moment. Without it, you are reconstructing the scene from limited evidence. The structured query logs also enable proactive optimisation: you can identify which data sources are consistently slow before they start causing failures.

The case against permanent additional logging: Additional logging does increase disk I/O on the gateway server, and the base64-encoded query text in the logs can contain sensitive data — column names, filter values, potentially PII depending on your query patterns. In environments with strict data classification requirements, storing query strings on a gateway server's local disk may violate your data handling policies. The log files also grow significantly under load — a busy gateway can generate several gigabytes per day under full additional logging, which requires a disciplined log rotation and retention strategy.

What I implemented: We enabled SystemCounterReportOn permanently — it captures resource utilisation with no query content, adding negligible overhead. QueryExecutionReportOn is enabled on a scheduled basis during peak refresh windows (typically 6am to 10am and during month-end batch runs), and disabled at other times via a scheduled PowerShell task that modifies the config file and restarts the gateway service. This gives us query-level telemetry when it matters most without accumulating sensitive query data continuously.

What Gateway Logs Actually Reveal

Gateway logs are not easy to read in raw form. They are stream-of-consciousness text files with inconsistent structure and significant noise. But once you build the parsing layer — or use the Performance template — they tell you things you cannot find anywhere else:

  • Which cluster member is handling which refresh — gateways distribute refresh requests across cluster members even for the same dataset, which is useful to know when a specific node has performance issues
  • Expired credential errors — GatewayErrors.log captures these the moment they occur, before users report failures, giving you a proactive window to act
  • Query execution patterns — which data sources are consistently slow, which queries are running 10x longer than their peers, which DirectQuery sessions are generating unexpectedly large result sets
  • Connectivity loss events — GatewayNetwork.log captures Azure Relay disconnections with timestamps, letting you correlate gateway outages with network events or upstream service incidents
  • Linking logs to the audit trail — the dataset ObjectId appears in the gateway info logs, allowing you to join gateway activity to the Power BI tenant activity log and trace a specific refresh failure all the way from the semantic model through the gateway to the data source query

The Architecture That Works at Scale

After deploying and managing this infrastructure across our enterprise, the configuration that has proven most resilient is:

  • A minimum 3-node Standard gateway cluster on separate physical hosts (not VMs sharing the same underlying hardware), each with identical GatewayCore.dll.config throttling thresholds
  • Load balancing enabled with CPUUtilizationPercentageThreshold set to 75 and MemoryUtilizationPercentageThreshold set to 80, preventing any single node from being saturated during peak refresh windows
  • Dedicated AD service account with least-privilege access to each data source, password rotation managed through a PAM solution
  • PowerShell monitoring that checks each cluster member's heartbeat every 5 minutes and alerts before the Power BI service even detects the outage
  • Permanent system counter logging, scheduled query logging during business hours, centralised log collection to a shared path accessible to the monitoring stack
  • Gateway installation policy enforcement that prevents Personal gateway proliferation and routes all new Standard gateway requests through change management

Gateway infrastructure is not glamorous. It sits below the semantic model layer, invisible to business users until it fails. That invisibility is exactly why it deserves this level of engineering attention — because when it fails, everything above it fails with it.
